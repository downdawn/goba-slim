# GoBA Slim 工程规则

## 开发边界

- 实现前先阅读当前公开需求、相关代码与配置。
- 文档和必要注释使用中文；代码标识、包名、配置键、环境变量、数据库对象和 API 字段使用英文。
- Gin 仅存在于 HTTP Transport 与 HTTP 平台边界；业务层不得依赖 Gin、pgx、sqlc 或 go-redis 的具体类型。
- OpenAPI 是 HTTP 契约事实来源，SQL 是查询事实来源；生成代码不得手工修改。
- 不使用运行时自动迁移。数据库变更只提供显式 SQL，由部署方手动执行。
- 不创建无边界的 `utils`、`helpers` 包；公共能力必须有稳定且跨模块的职责。
- 模块与文件按实际职责拆分，不预建空抽象或空目录。

## 安全与质量

- Secret、密码、Token、Cookie、私钥和 Authorization 信息不得写入日志、错误响应、示例配置或镜像。
- 新依赖需确认维护状态、安全公告、许可证与标准库替代方案。
- 行为变更必须同步测试、公开文档和示例配置。
- 可自动检查的规则应交由 CI 执行。
- 提交信息使用简洁英文 Conventional Commits。

## 验证

- 修改 Go 代码后至少执行相关包测试、`go test ./...`、`go vet ./...` 与格式检查。
- OpenAPI 变更后重新生成代码并检查无漂移。
- 本机无法执行的 Docker、Race 或外部服务验证必须如实记录，由 CI 或目标环境补验。
