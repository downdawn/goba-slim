# 路线图

GoBA Slim 按可独立运行、测试和评审的阶段演进。每个阶段必须保持主分支可构建，并同步契约、测试、公开文档和示例配置。

## Phase 1：工程内核

**状态：已完成。**

提供强类型配置、结构化日志、统一错误、安全 HTTP 边界、模块生命周期、健康检查、OpenAPI、CLI、测试、CI 和非 root 容器镜像。不包含 PostgreSQL、Redis、用户、认证、文件或动态配置的运行时实现。

## Phase 2：PostgreSQL 与用户模块

**状态：已完成。**

已建立 PostgreSQL 连接与就绪检查、显式 SQL 初始化和版本检查、sqlc 查询边界、事务抽象及用户模块。验收覆盖真实 PostgreSQL 集成测试、管理员初始化、用户核心能力、事务回滚和并发唯一约束。

Phase 2 原定不开放用户 HTTP 接口；相关路由已在 Phase 3 认证边界建立后接入。

## Phase 3：Redis 与认证会话

**状态：已完成。**

已建立 Redis 用途级接口、登录、短期 EdDSA Access Token、可撤销 Refresh Token、会话生命周期和按 `kid` 的公钥轮换。集成验收覆盖登录、当前用户、用户列表、刷新、Token 重用撤销、登出、并发刷新、改密即时失效、停用即时失效及 Redis 故障 fail closed。

## Phase 4：可选文件模块

**状态：已完成。**

提供可配置启停的文件能力，覆盖流式上传、大小限制、内容识别、安全对象 Key、存储边界和路径输入测试。模块关闭时不得注册相关路由。

## Phase 5：可选动态配置模块

**状态：已完成。**

提供非 Secret、类型化的业务配置和缓存失效机制。Secret、信任边界和启动关键配置不得通过运行时接口修改。

## Phase 6：生产交付

**状态：已完成。**

完善诊断、集成测试、Secret 与镜像扫描、多平台构建、SBOM、镜像签名、部署示例和版本发布流程。功能是否进入该阶段，以真实部署需求和维护成本为依据。

## 非目标

- 不兼容既有 FBA API、数据库或前端。
- 不在运行时从 Git、ZIP 或 Go 插件动态执行代码。
- 不引入全局数据库、Redis 或字符串 Service Locator。
- 不为尚未实现的模块预建目录和空抽象。
- 不在服务启动时自动修改数据库结构。

架构边界见 [`architecture.md`](architecture.md)。
